# Hermes Config Migration Preservation Gate

> Draft generated by researcher tick34 after GH #62723. Review before installation.

## Safety boundary

This procedure is read-only against production profiles until explicit switch-over. Run migrations on copies. Do not write production `config.yaml` during verification.

## Trigger

- `_config_version` changes;
- `hermes config migrate` or setup rewrites config;
- upgrade crosses a schema boundary;
- more than one profile is present;
- any platform adapter disappears after upgrade.

## Workflow

### 1. Inventory

Enumerate default plus every named profile. For each config, record:
- relative path;
- schema version;
- top-level key set;
- nested key paths under `platforms`, `model`, `agent`, `gateway`, `kanban`, `toolsets`, `fallback_providers`;
- count of configured platform adapters;
- hash of the full file.

Values matching authentication fields must be masked in reports.

### 2. Snapshot

Create a timestamped copy outside the live config path. Verify every copied file by hash. Record permissions and owner.

### 3. Offline migration

Run the exact migration binary/version against a temporary Hermes home containing the copied profile tree. Never point the migrator at the production home for the first pass.

### 4. Structural diff

Compare before/after key paths with these rules:
- documented rename → allowed only with value equivalence;
- documented deletion → allowed only with explicit replacement;
- `platforms` section or adapter count decrease → hard fail;
- non-empty value becoming empty/null → hard fail unless the migration spec explicitly requires it;
- unknown top-level key deletion → hard fail.

### 5. Semantic probes

For each profile:
- parse YAML;
- resolve the expected active model/provider;
- list configured platform adapters without starting them;
- check disabled toolsets and approval mode;
- verify profile-local paths remain profile-local.

### 6. Roundtrip

Serialize the migrated config once more through the current config writer and rerun the structural diff. This catches writers that parse correctly but drop unknown sections when saving.

### 7. Canary readiness

Start only an isolated canary copy if the environment supports it. Require:
- expected adapter count;
- API listener ready;
- heartbeat advancing;
- no config rewrite on startup.

### 8. Switch and readback

Only after all profiles pass, switch production. Immediately read back each config and compare key manifests and hashes to the verified migrated copy.

### 9. Rollback

On any mismatch:
1. stop the affected gateway cleanly;
2. restore snapshots preserving owner/mode;
3. read back hashes;
4. start gateway;
5. verify adapter readiness.

## Fail conditions

- any `platforms` block disappears;
- adapter count decreases unexpectedly;
- a stored value becomes empty;
- migration changes a different profile;
- backup hash cannot be verified;
- gateway process starts but adapter/API/heartbeat readiness is incomplete.

## Evidence

Seed incident: GH #62723 reported v30→v32 migration removing `platforms.feishu` from 9 of 10 profiles while gateways kept running without adapters.
