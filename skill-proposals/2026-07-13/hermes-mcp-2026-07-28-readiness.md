# Hermes MCP 2026-07-28 Readiness

> Draft generated by researcher tick34. The final specification is scheduled for 2026-07-28. Until final publication, run probes against isolated/canary environments only.

## Scope

This skill prepares a Hermes MCP integration for the draft revision while keeping the legacy 2025-era path operational.

## Major contract changes

- protocol-level sessions and `Mcp-Session-Id` are removed;
- `initialize` / `initialized` are removed for the modern era;
- every request carries protocol/client metadata;
- `server/discover` advertises versions/capabilities;
- Streamable HTTP standard headers must agree with the body;
- cacheable list/read results carry TTL and cache scope;
- server-initiated interactions become request-scoped multi-round trips;
- roots, sampling, logging, and HTTP+SSE are deprecated;
- authorization adds issuer and resource binding requirements;
- tool schemas support JSON Schema 2020-12 and require resource bounds.

## Workflow

### 1. Freeze assumptions

Document current behavior:
- SDK version;
- negotiated protocol version;
- transport;
- use of session IDs/handshake/ping;
- list caching/invalidation;
- OAuth issuer/resource handling;
- use of deprecated features.

### 2. Add explicit era selection

Keep legacy as the production default. Add modern support behind an explicit opt-in. Never infer modern behavior solely from an SDK package upgrade.

Expected modes:
- legacy: current handshake;
- auto: probe and fall back to legacy;
- pinned modern: fail if the peer cannot negotiate the modern revision.

### 3. Implement discovery

Probe `server/discover` and validate:
- supported protocol versions;
- server identity;
- capabilities/extensions;
- deterministic failure against legacy-only peers.

### 4. Stateless request validation

For each modern request:
- protocol version, client info, and capabilities are present in request metadata;
- no hidden dependency on a prior handshake or sticky server instance;
- cross-call state uses explicit opaque handles in tool arguments;
- retries use a new request ID after a broken response stream.

### 5. Header/body consistency

Verify required standard headers match the JSON-RPC method/tool name. Missing, malformed, or mismatched headers must be rejected before tool execution.

### 6. Cache isolation

For `tools/list`, `prompts/list`, `resources/list/read`, and templates:
- require TTL and cache scope;
- default unknown policy to TTL 0 and private scope;
- never share private results across users/profiles;
- keep deterministic list ordering.

### 7. Authorization binding

Verify:
- authorization response issuer exactly matches the recorded issuer when present;
- stored client credentials are keyed by issuer;
- credentials are never reused after an authorization-server change;
- client sends the target resource indicator;
- server validates token audience;
- inbound access material is never passed through unchanged to an upstream API;
- PKCE S256 support is verified before authorization.

Reports must mask all auth material.

### 8. Deprecated feature inventory

Flag new usage of:
- Roots;
- Sampling;
- MCP Logging;
- HTTP+SSE;
- deprecated includeContext modes.

Provide a migration target: tool parameters/resource URIs, direct provider integration, stderr/OpenTelemetry, and Streamable HTTP.

### 9. Tool schema resource bounds

Support JSON Schema 2020-12 while enforcing:
- maximum depth;
- bounded validation time;
- no automatic external reference fetch;
- bounded reference expansion;
- size limits on schemas and structured results.

### 10. Dual-era conformance matrix

| Scenario | Legacy | Modern opt-in |
|---|---|---|
| connect/discover | | |
| list tools | | |
| call tool | | |
| header mismatch rejection | N/A | |
| private cache isolation | | |
| multi-round trip input | | |
| auth issuer mismatch | | |
| resource audience mismatch | | |
| broken stream retry | | |

### 11. Rollout gate

Do not make modern the default until:
- final spec is published;
- official SDK support is stable;
- all matrix cells pass;
- downgrade/legacy fallback works;
- no production profile loses MCP tools.

## Pitfalls

- Installing a v2 SDK does not necessarily put modern bytes on the wire.
- Stateless protocol does not mean the application cannot have explicit state handles.
- Process/session readiness is not protocol conformance.
- Cache scope errors can become cross-profile data exposure.
